# Implementation Plan: Real-Time Appointment Booking

**Branch**: `014-realtime-booking` | **Date**: 2026-06-03 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/014-realtime-booking/spec.md`

---

## Summary

Implement end-to-end real-time appointment booking by introducing a shared Redis appointment hash cache between the API and Worker services, a per-resource occupancy sorted-set structure for technicians and service bays, three new API endpoints for querying active appointments and occupied slots, and UI polling logic in the `api-client-ui` create-appointment modal. The Worker is extended to maintain the occupancy structures on `Scheduled` and `Completed/Cancelled` transitions.

---

## Technical Context

**Language/Version**: TypeScript 5.x (API + UI), C# .NET 8 (Worker)

**Primary Dependencies**:
- `ioredis` (Node.js Redis client, already in use)
- `StackExchange.Redis` (C# Redis client, already in use)
- `React 18 + Vite` (UI, already in use)
- `Zod` (API validation, already in use)
- `FluentValidation` (Worker, already in use)

**Storage**: Redis (shared cache + streams, already provisioned), PostgreSQL (Worker persistence, already provisioned)

**Testing**:
- Node.js: `jest` (unit), Docker Testcontainers via existing test setup (integration)
- C#: `xUnit` + `dotnet test`, Docker Testcontainers (integration)

**Target Platform**: Docker Compose (all services)

**Project Type**: Cross-service feature touching three existing applications

**Performance Goals**: Active appointments endpoint < 500 ms p95; occupancy endpoint < 500 ms p95

**Constraints**: No new services permitted by constitution; all new Redis operations must go through Port interfaces; `KEYS` command forbidden (use `SCAN` or index sets)

**Scale/Scope**: Per-tenant scoping; designed for up to several hundred active appointments per tenant
