## 1. Consistent Hashing Function `f(tenant_id, vehicle_id) → partition_id`

**Decision**: Use FNV-1a 32-bit hash over the concatenated string `{tenant_id}:{vehicle_id}`, then modulo N (default 4).

**Rationale**:
- FNV-1a is deterministic, has zero external dependencies, runs in O(1), and produces well-distributed output for UUID-shaped inputs.
- Modulo N gives a stable partition in `[0, N-1]` across any number of service instances because the function is pure — no shared state.
- The seed and algorithm are fixed in code, not configurable, to prevent accidental partition drift.
- Node.js has no built-in FNV; implement with a ~10-line pure TypeScript function in `src/domain/utils/partition-hash.ts` — no npm package needed.

**Alternatives considered**:
- `crypto.createHash('sha256')` — deterministic but overkill; SHA-256 is ~10× slower than FNV-1a for short strings and returns a 256-bit value requiring BigInt modulo.
- `murmurhash` (npm) — excellent distribution but adds a dependency; FNV-1a is simpler for this cardinality.
- `Math.random()` / UUID-based — NOT deterministic; rejected immediately.

## 2. Concurrent Request Handling (FR-013 / SC-004)

**Decision**: Use Node.js `cluster` module to fork one worker process per CPU core. Each worker runs the full Express app independently. Shared state (Redis) is accessed via the process's own `ioredis` connection pool.

**Rationale**:
- Node.js is single-threaded; the event loop handles I/O concurrency well but CPU-bound blocking on a single core can cause head-of-line blocking under high load.
- The `cluster` module is built into Node.js, requires no additional npm packages, and mirrors the constitution's preference for minimal dependencies.
- Because all state is in Redis, clustering is share-nothing — no in-process synchronisation primitives needed.
- `PM2` is an alternative but adds a runtime dependency; the native `cluster` approach is suitable for the Docker-based deployment model (one container per replica).

**Alternatives considered**:
- Worker threads (`worker_threads`) — suitable for CPU-bound parallelism, not I/O. Overhead of message passing is unnecessary here.
- Multiple container replicas behind a load balancer — valid for production scale-out but doesn't eliminate the single-process bottleneck inside each container; cluster + replicas is complementary.
- `fastify` + async pipeline — still single-process without cluster.

**Implementation note**: `src/infrastructure/http/cluster.ts` acts as the entry point. It forks `os.cpus().length` workers. Each worker imports `src/infrastructure/http/server.ts` (Express app factory). The `--max-old-space-size` Node.js flag is set via `Dockerfile` environment to bound memory per worker.

## 3. Appointment Hash — Redis Data Structure Choice

**Decision**: Store the appointment hash as a Redis Hash (HSET) with field-per-attribute layout under the key `tenant:{tenantId}:appointment:{vehicleId}`.

**Rationale**:
- A Redis Hash allows the worker to atomically update individual fields (e.g., set `technicianId` and `bayId`) when they become known, without overwriting the whole record.
- A JSON string (SET) requires a full read-modify-write cycle, which creates a TOCTOU race condition in the worker.
- The existence check at the API layer is a single `EXISTS` command — O(1) regardless of hash size.
- No TTL is set on the key (spec FR-005). The worker deletes the key when it finishes processing.

**Fields in hash at intake time**:

| Field | Value | Notes |
|---|---|---|
| `commandId` | UUID v4 | Generated by API service |
| `tenantId` | string | From validated payload |
| `customerId` | string | From validated payload |
| `vehicleId` | string | From validated payload |
| `serviceTypeId` | string | From validated payload |
| `desiredStartTime` | ISO 8601 | From validated payload |
| `source` | `"admin"` | Hardcoded for this endpoint |
| `timestamp` | ISO 8601 | Server-side intake timestamp |
| `technicianId` | (absent) | Null at intake; worker sets later |
| `bayId` | (absent) | Null at intake; worker sets later |
| `workshopId` | (absent) | Null at intake; worker sets later |
| `partitionId` | number 0–3 | Computed by hash function |

## 4. Zod Validation Schema Design

**Decision**: Single `CreateAppointmentCommandSchema` with all fields required at the HTTP boundary (FR-003). `desired_start_time` validated as ISO 8601 future datetime.

**Rationale**:
- Constitution §IV mandates Zod for all HTTP boundary validation in Node.js.
- Zod's `.refine()` method handles the future-date guard cleanly without external date libraries.
- The schema lives in `src/application/commands/create-appointment.command.ts` as a Zod schema + inferred type export, making it importable in both the controller and the use case input model.
- `z.string().uuid()` enforces UUID format on `tenant_id`, `customer_id`, `vehicle_id`, and `service_type_id`.
- `z.string().datetime()` + `.refine(v => new Date(v) > new Date())` enforces ISO 8601 future-only.

## 5. Redis Client — ioredis Configuration

**Decision**: Single `ioredis` client instance shared across the process via a module-level singleton. Configured with `lazyConnect: true` and a connection retry strategy.

**Rationale**:
- Constitution tech stack table mandates `ioredis`.
- The client is wrapped behind an `ICacheProvider` port interface defined in `src/application/ports/cache-provider.port.ts` to maintain layer boundaries (clean_architecture rule §1).
- A concrete `RedisCacheAdapter` in `src/infrastructure/cache/redis-cache.adapter.ts` implements `ICacheProvider` using `ioredis` internally.
- Infrastructure-level `ReplyError` / `AbortError` from ioredis are caught inside the adapter and re-thrown as `CacheUnavailableException` (domain exception) — per the Portability Rule in constitution §II and clean_architecture rule §2.
- The same pattern applies for the Redis Stream publisher (`IMessagePublisher` port → `RedisStreamPublisher` adapter).

## 6. Tenant Context Propagation — AsyncLocalStorage

**Decision**: Initialise one `AsyncLocalStorage<TenantContext>` instance at app startup. The Express middleware reads `tenant_id` from the validated request and calls `store.run({ tenantId })` before passing to the next handler. All downstream code reads from the store.

**Rationale**:
- Constitution §III (and multi_tenancy rule §1) explicitly prohibits parameter-drilling of `tenant_id`.
- `AsyncLocalStorage` is built into Node.js ≥ 12.17.0 — no additional packages required.
- The store is defined in `src/domain/context/tenant-context.ts` as a pure export — no framework dependency.
- The middleware populating the store lives in `src/infrastructure/http/middleware/tenant-context.middleware.ts`.

## 7. Health Check Implementation

**Decision**: `GET /health` pings Redis with `PING` command. Returns `200 { status: "ok" }` on success, `503 { status: "error", reason: "redis_unavailable" }` on failure (SC-006).

**Rationale**:
- Simplest live Redis check; does not require a full write/read cycle.
- Kubernetes liveness/readiness probes and Docker Compose health checks both work with a simple HTTP status code.
- Route is in `src/infrastructure/http/routes/health.routes.ts`, calls a `HealthCheckUseCase` in the Application layer that invokes `ICacheProvider.ping()`.
