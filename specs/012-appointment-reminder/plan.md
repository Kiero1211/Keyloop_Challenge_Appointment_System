# Implementation Plan: Appointment Reminder

**Branch**: `012-appointment-reminder` | **Date**: 2026-06-02 | **Spec**: [spec.md](file:///Users/kiero/Desktop/Code/Keyloop/specs/012-appointment-reminder/spec.md)

**Input**: Feature specification from `/specs/012-appointment-reminder/spec.md`

## Summary

Implement a daily scheduled background task in the C# worker service that queries a newly created PostgreSQL view (`AppointmentReminderView`) to identify appointments starting within 48 hours. For each identified appointment, it triggers an email notification and records the reminder as sent to prevent duplicates, strictly adhering to tenant isolation.

## Technical Context

**Language/Version**: C# .NET 8, PostgreSQL, SQL

**Primary Dependencies**: `Microsoft.Extensions.Hosting` (for `BackgroundService`), Entity Framework Core (for DB view mapping and migration), and potentially an email sending Port.

**Storage**: PostgreSQL (adding a view and potentially a flag/table for tracking sent reminders).

**Testing**: xUnit, Moq, Testcontainers (for C# worker service).

**Target Platform**: Linux server / Docker container (Worker Service).

**Project Type**: Background Worker Service (`apps/appointment-worker-service`).

**Performance Goals**: Efficient querying via `AppointmentReminderView` to handle thousands of daily appointments without performance degradation.

**Constraints**: Must strictly maintain multi-tenancy boundaries (querying and processing must not mix tenant data improperly) and ensure idempotency (exactly one email per appointment).

**Scale/Scope**: Impacts the worker service and database schema only.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **Hexagonal Architecture**: The scheduled task fits perfectly in the worker service, which owns background processing and database interactions. It will not expose HTTP endpoints. (Pass)
- **Clean Architecture**: We will define an Application Layer Port (e.g., `IAppointmentReminderUseCase` or `IEmailService`) and implement the scheduled task as an Infrastructure hosted service. (Pass)
- **Multi-Tenancy**: The database view MUST include `tenant_id`. The worker will process reminders and include `tenant_id` in any dispatched events or email flows. (Pass)
- **TDD**: Tests will be written before implementation. (Pass)
